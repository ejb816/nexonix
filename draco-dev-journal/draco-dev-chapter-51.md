## 1. Assembly — wiring actor groups as pure data

The session opened on the queued thread:

> **Dev:** I'm excited to continue with the assembly of actors, unless there is something else that should be done first.

The design (ch.50's sketch) became code. `draco.Binding` (`from`/`param`/`to`) and `draco.Assembly` (`members: Seq[TypeName]`, `bindings: Seq[Binding]`, `entry: TypeName`) are ordinary generated draco-core types — no Pekko import. Topology is addressed by **identity (TypeName)**, never `ActorRef`. Two properties carry the design:

- **`AssemblyValidator`** (`src/mods`, no Pekko): per binding, the `from` member's factory must declare `param`, typed `ActorRef[M]`, and `M` must equal the `to` member's `Actor[M]` message type. Reads only TypeDefinitions.

The one thing it can't do generically — call a member's typed `actorType(...)` — is supplied by the caller as a per-member `Constructor` adapter.

First proof: the Aerial `Creator → Consumer` chain (stateful Consumer reaping at PostStop) as an Assembly — validates without Pekko, rejects a mis-wired binding, runs identically to the hand guardian. Green, 192/192.

## 2. Primes' real goal — combinatorial induction

A pivot toward the declarative language stalled long enough for the Dev to disclose what Primes is actually for:

> **Dev:** What I am going for, but have not achieved, is a working example of "combinatorial induction". This process would start with the syntactic rules for generating Prime Ordinal Notation (PON) sequence from the first two primes p1 (2) and p2 (3) and then generating: 1. The binary form of the cardinal values `2 = (:.), 3 = (::), 4 = (:..)` 2. The PON expression for the subsequent numbers `2 = (), 3 = (()), 2^2 = 4 = (:())` 3. The Goldbach 2-sums and 3-sums added combinatorially from existing primes

Decoded: `:` = binary 1, `.` = binary 0 (so the cardinal layer is the number in base 2). PON is a recursive prime-power notation. The Dev then pinned the rule:

> **Dev:** For ANY PON `(...)`, `...`= index of a prime power. If power > 1 and index > 1 then `(index:power)`; if index = 1 then `(:power)`; if power = 1 then `(index)`. For ALL PON index = product of powers of primes.

— with index and power *themselves* recursively PON (forced by `4 = 2² = (:())`), bottoming out at `PON(1) = «empty»`. And after a `Squirrel!` interlude:

> **Dev:** Composite numbers are just adjacently concatenated prime powers in ascending base index order (2^2 comes before 3)

So the whole notation is two rules: a prime power renders `(index∶power)` (colon dropped at power 1, index empty at 1, both recursive); a composite is its prime-power factors juxtaposed in index order. PON is itself a term-language generated by induction over its own grammar — Primes-done-right is a microcosm of "draco in draco."

## 3. Mechanizing PON

The Dev shared a comprehensive 0..79 corpus (`Prime ordinal 78`) with `0 = "."` (the addition operator / additive identity), `1 = ":"`, and `.` as the infix sum for Goldbach n-sums. To prove comprehension, the move was to *mechanize* it. A Python evaluator drew a sharp correction:

> **Dev:** Write it using type definitions. If that's too vague then use Scala instead of Python.

(Recorded as [[stay-in-draco-not-python]].) The Scala `Pon` evaluator (a sealed `Zero/One/PrimePower/Product/Sum` family — the seed of a draco type-def family) validated against the agreed values, then ran the corpus: **42 discrepancies of 550** — ~18 paren-drop typos (one recurring: `((:(()))` for `((:(())))`=19), one canonical typo (N=40), one stray product under N=23, and a cluster of misplaced 75-block rows under N=73. A reading prompt also surfaced: a test reading `~/Downloads` tripped a macOS TCC consent dialog — fixed by copying the corpus into `src/test/resources` and reading from the classpath (no out-of-repo reads).

Rather than hand-patch the error-prone document, the Dev chose to build the **encoder** (`Pon.encode`/`render`/`canonical`) — the inverse of the evaluator and the actual generative engine. With the Goldbach policy set to **exhaustive** (all 2-prime sums for even N, all 3-prime sums for odd, plus `2+(N−2)`), `Pon.representations` regenerates a correct-by-construction oracle; the document is now emitted from the encoder rather than maintained. The relationship inverted: the generator is the source of truth, the document a derived rendering.

## 4. The monad insight

Mid-design, a reframe worth recording:

> **Dev:** "A monad is a monoid in the category of endofunctors." Working memory is a monad. Accumulator is a monad. There can be logical discipline for syntactically accessing monads in a framework that supports operational execution.

> **Dev:** I added Monadic just so I could have an escape hatch for operational expressions.

This overturned a sloppy framing of mine ("convergence = empty the Monadic"). `Monadic` is named literally: the operational substrates *are* monads — the `Accumulator` is Writer over set-union monoids (with `primeSet` as a State sieve), working memory is the fact-effect monad. Convergence is **disciplining** the Monadic (lawful, typed ops over declared monads), not eliminating it. The language's third register — disciplined effects — is therefore still "logic" (equational law), not a lawless hatch.

## 5. DRAKE — the language, and a notation odyssey

> **Dev:** I'm considering naming the draco native language drake (domain rules actor knowledge engine).

DRAKE (`.drake`) it is — `draco` (dragon/engine) ↔ `drake` (its native tongue), acronym mapping the pillars. A foundational boundary was set by the Dev against my over-reach toward a "higher" / programming-language surface:

> **Dev:** JSON is the canonical and normative data format for type definitions, as well as the authoritative source for the Generator. You will have to explain why I should go "higher" to make it syntactically closer to a conventional programming language.

I conceded: with JSON normative, DRAKE is an L0 authoring surface, transparent to JSON; "program-like" both undercuts "JSON is canonical" and *fights* the non-programmer goal (a domain expert wants structured data/logic, not C-syntax). The only legitimate "higher" is the normative *model* structuring logic-as-data — a separate decision.

**The bracket convention took several reversals** (preserved as the abandoned paths):

> **Dev:** I suggest using "()" for hetero-type-tuples (both type and value) and leave "[]" for homo-type sequences.

> **Dev:** You did just the opposite of what I suggested. I was looking for Seq[Variable] -> Seq(Variable) … Look past my ad hoc conflation of scala and drake expression syntax and try to discern my intent.

> **Dev:** I want to KEEP "[]" for homo-type sequences and use "()" for hetero-type tuples

> **Dev:** But elements is followed by a homo-type sequence of TypeElement so it should be enclosed (in drake) by "[]" and the Seq symbol should acknowledge the hetero-type nature of multiple type parameters (even if there is only one parameter) and, in drake, use "()" to enclose the type parameter(s).

> **Dev:** Now that I see it, what about Seq(T) -> [T]?

> **Dev:** How about using "{}" for Set the way we use "[]" for Seq?

The settled taxonomy: **`[T]`/`[a,b]` = homogeneous ordered sequence (Seq); `{T}`/`{a,b}` = homogeneous unordered set (Set); `(A,B)`/`F(A,B)` = heterogeneous tuple AND type-application.** This is Haskell's convention (`[a]`, `(a,b)`) — familiar, and it makes the brackets *mean* something.

Then `mut`:

> **Dev:** Is there a way of using "mut" that would be consistent with how it [i]s used as a body element?

Resolved as a single left-modifier "mutable": leading position → mutable binding (`var`); type position → mutable collection (`mut {Int}` = `mutable.Set[Int]`). The body-element `mut` falls out as the field-level instance of the rule.

> **Dev:** Ooops, I misled you. I meant if the second kind was elided, it would mean that it is, by default, immutable (not fixed).

— so **elided `mut` = immutable (default)**; binding kinds stay explicit (no fix-default — which is *necessary* because the metamodel has fields literally named `body`/`parameters` that would collide with block keywords).

The metamodel was then hand-authored in DRAKE as the future parser's test corpus: `Action`, the full `TypeElement` family, `TypeName`. And a pacing directive landed:

> **Dev:** Keep it simple and incremental. I sometimes lose track of conceptual threads when there are significant quantitative leaps.

(Recorded as [[small-increments-one-concept]].)

> **Dev:** TypeName does double duty … we should investigate whether we should refactor TypeName so that it's part of the TypeElement family and some of its complexity allocated to Generator.

After fixing an IntelliJ-mangled `f"…"` in `PonCorpusTest` and re-adding the canonical trailing blank line, **DracoGenTest 103/103**. Path A + the `domainPackage` rename are deferred and bundled (both wide-surface). DRAKE now needs only one construct set — no `dyn`, no `globalElements`.
