**Low confidence** ‚Äî I'd recommend against it right now. Here's why:

The Generator produces the structural skeleton well (trait + companion with `typeDefinition`/`typeInstance`/`apply`/`Null`/`Default`), but both files have significant hand-written content that the Generator doesn't yet cover:

**TypeDefinition.scala gaps:**

- The entire encoder/decoder dispatch (lines 72‚Äì175) ‚Äî the `kind`-based discriminated union codec is hand-written
- Custom factory body logic ‚Äî e.g., `Dynamic.apply` hardcodes `value = ""`, `Action.apply` hardcodes `name = "ctx"`, `Condition.apply` hardcodes `valueType = "Boolean"`. These aren't captured in the Factory parameters

The generated output would be a valid starting point, but replacing the hand-written files would break compilation immediately due to the missing codecs and custom factory logic. These files are good candidates for generation *after* the Generator gains support for codec generation, import generation, and richer factory body specifications.

## Generator Roadmap Discussion

**Draco:** It's the right direction, but looking at the priority list in memory, I think it's premature as the *immediate* next step. There are prerequisites:

1. **Import generation (items 2 & 3 on the priority list)** ‚Äî Codec generation emits code that references `Encoder`, `Decoder`, `Json`, `EncoderOps`, etc. Without import generation, the generated codec code won't compile. Import generation is a simpler, more contained task that unblocks codec generation.

2. **Consistency updates (item 1)** ‚Äî Locking in the TypeInstance/DomainInstance companion object pattern ensures the Generator's output structure is stable before layering codecs on top.

The natural sequence would be:

1. Consistency updates (already planned)
2. Import generation ‚Äî package-path-based + framework-conditional (Circe imports when codecs are present)
3. **Then** codec generation ‚Äî which itself has real complexity:
   - Simple field-based encode/decode (TypeDefinition, TypeName) ‚Äî relatively mechanical
   - Discriminated union dispatch (TypeElement's `kind`-based encoder/decoder) ‚Äî the hard part
   - Sub-type codec wiring (`Codec.sub[Parent, Child]`) on leaf companions
   - Custom factory body overrides (hardcoded fields like `name = "ctx"`) need richer Factory metadata first

## Planning Consistency Updates

> **Dev:** yes *(in response to: "Ready to start planning step 1 ‚Äî consistency updates?")*

**Draco:** *(Enters plan mode. Launches an Explore agent to thoroughly analyze companion object patterns across the codebase ‚Äî TypeInstance, DomainInstance, RuleInstance ‚Äî reading 20+ files. Then reads PrimesFromNaturalSequence.scala, Base.scala, Primes.scala, DomainInstance.scala, DomainName.scala, Domain.scala, DomainType.scala, and domain JSON files.)*

### Key Findings

**TypeInstance pattern** (every framework companion): `extends App with TypeInstance`, `lazy val typeDefinition`, `lazy val typeInstance: Type[X]`, optional `apply`/`Null`/`Default`/codecs.

**DomainInstance pattern** (Base, Primes): `extends App with DomainInstance`, same typeDefinition/typeInstance, plus `lazy val domainInstance: Domain[X]` with DomainName and TypeDictionary. Two sub-patterns found ‚Äî Base uses inline `new Domain[Base] { ... }`, Primes uses the cleaner `Domain[Primes](_domainName = ...)` factory call.

**RuleInstance pattern** (generated rules): `extends App with RuleInstance`, private ruleDefinition + condition functions + action/pattern + `lazy val ruleInstance: RuleType`.

### Three Inconsistencies Identified

1. **Factory body** ‚Äî Generator's `factoryBody` ignores `Factory.body` field, derives overrides purely from parameters. Hand-written types like Action, Condition, Pattern have custom overrides (`override val name = "ctx"`, `override val valueType = "Boolean"`) not derivable from parameters. Factory already has a `body: Seq[BodyElement]` field designed for this ‚Äî just unused.

2. **Null generation** ‚Äî Generator produces `null.asInstanceOf[X]` for every field. Hand-written code calls the factory: `TypeDefinition(TypeName.Null)`, `Condition(_parameters = Seq.empty, _value = "")`.

3. **No DomainInstance generation** ‚Äî Generator handles TypeInstance and RuleInstance but not DomainInstance.

### Plan Designed

*(Launches a Plan agent to design the implementation. Writes the plan to the plan file.)*

**Step 1: Refactor `factoryBody`** ‚Äî New signature takes `Factory` instead of `Seq[Parameter]`. When `factory.body` is non-empty, use body entries directly as override lines. When empty, fall back to parameter-derived behavior (including automatic `typeDefinition` override).

**Step 2: Refactor `nullInstance`** ‚Äî When factory exists, call `apply()` with null-equivalent args via new `nullValueFor` helper. Fall back to null-cast only when no factory exists. Helper handles String ‚Üí `""`, Seq ‚Üí `Seq.empty`, Int ‚Üí `0`, Boolean ‚Üí `false`, unknown ‚Üí `null.asInstanceOf[T]`.

**Step 3: Add DomainInstance generation** ‚Äî New `generate(td: TypeDefinition, dn: DomainName)` overload targeting the Primes pattern using `Domain.apply`. New private helpers `domainGlobal` and `domainInstanceLiteral`.
